# Benchmark Testing & Reproducibility (B1 / QE4.2)

This document defines the gates and reproducibility contract for the comparative
benchmark. All of it is consolidated, with results, into the standalone
**Test Report** (`prototype/reports/TEST-REPORT.md`) that becomes a thesis
appendix.

## 1. Correctness gate (runs BEFORE timing is trusted)

A latency comparison is only valid if every backend computes the **same answer**
on the same input. For each (backend × query × payload) cell the harness checks
the backend result against the plaintext oracle (`workload.referenceResult`)
via `withinTolerance`:

- Exact queries (`sum`, `max`, `quantile`, and integer `mean`) must match exactly
  (relative tolerance 1e-6).
- FHE approximate-numeric (CKKS) results are compared within a stated tolerance;
  the tolerance used is recorded in the Test Report.

If a cell fails the gate, its timing is reported but flagged `correct=false` and
excluded from headline comparisons.

## 2. Reproducibility contract

- **Fixed seeds:** datasets are generated by a deterministic PRNG (`workload.ts`,
  seed 42) so every backend sees identical data and reruns are stable.
- **Pinned environment:** record exact CPU/RAM/OS, Node version, OpenFHE version,
  SGX/Azure-CC mode, and dependency versions in the Test Report.
- **One-command run:** `npx tsx prototype/benchmarks/src/harness.ts` (env vars
  `BENCH_REPS`, `BENCH_PAYLOADS` documented in `README.md`).
- **Raw data committed:** `results/bench_raw.csv` holds every replicate, not just
  the summary, so the statistics can be re-derived.

## 3. Output artifacts

- `results/bench_raw.csv` — one row per replicate (backend, query, payload, rep, latency_ms, correct).
- `results/bench_summary.csv` — one row per cell (mean ± 95% CI, min/max, indicative RSS, correctness).
- Figures 1–3 (latency CI bars, throughput × concurrency, stress heatmap) are
  generated from these CSVs by the thesis figure pipeline.

## 4. Trust-assumption column (qualitative, stated per backend)

Latency alone is not comparable across backends without stating what each one
*assumes*: `our-protocol` = information-theoretic (Shamir t-of-n, shares reveal
nothing); `fhe` = computational (lattice hardness); `tee` = hardware-vendor
(enclave manufacturer); `centralized` = none (plaintext). This column appears in
every results table so a fast-but-weaker trust model is never mistaken for a win.

## 5. Threats to validity

- **JIT / GC jitter:** mitigated by ≥10 replicates + CI; warm-up not discarded
  (documented). Pin the machine; avoid background load.
- **Memory figures:** the harness records an *indicative* RSS snapshot only; the
  real per-backend memory profile (especially the FHE/TEE memory wall noted in
  `docs/dsr-5.md`) is measured directly in Tier 1.
- **Hardware dependence:** TEE results depend on the SGX/Azure-CC mode; stated explicitly.
- **Scope:** the benchmark measures the fixed workload only; it is not a claim
  about arbitrary workloads or production deployment.
